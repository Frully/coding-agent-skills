# Operating Git Worktrees

使用 `git worktree` 实现多 agent 并行开发的指南。命令级操作说明见 [SKILL.md](./SKILL.md)。

## 什么是 worktree，为什么要用它

Git worktree 是同一个仓库的另一个工作目录——共享同一份历史和对象存储，但在不同的目录中检出不同的分支。可以把它理解成：不用重新 clone，也能得到这个仓库的另一份独立工作区。

最核心的动机是**多 agent 并行开发**。每个 agent 拥有自己的 worktree 目录和自己的分支，多个 agent 可以同时开发而互不污染。主工作区始终稳定停留在 `main`、`develop` 或你使用的基线分支上，实际的任务开发全部在辅助 worktree 中进行。

一个 agent、一个目录、一个分支——互不干扰。

## 前提条件

- Git 2.15 或更高版本（worktree 的关键改进从此版本开始）
- worktree 根目录（`.worktrees/`）应写入 `.gitignore`，避免辅助工作区被纳入版本控制

## 基本使用思路

保留一个主工作区，让它持续停留在常用基线分支上。

真正的任务开发放到额外的 worktree 中进行。默认约定将辅助 worktree 放在 `<repo>/.worktrees/` 下。

推荐的心智模型是：

- 主工作区：始终停留在基线分支
- 辅助 worktree：用于具体任务分支
- 任务集成完成后：移除或复用这个辅助目录

## 两种不同的工作流

worktree 常见有两种组织方式，它们适合的场景不同。

### 基于任务的工作流

在这种方式里，每个任务都有自己独立的 worktree。

典型结构：

```text
repo/
├── .worktrees/
│   ├── login-page/
│   ├── fix-timeout/
│   └── billing-copy-update/
└── <主工作区>
```

适合这些情况：

- 希望任务和目录一一对应，最清晰
- 一个任务还没结束时，另一个任务也可能同时推进
- 不介意随着任务创建和移除 worktree

工作流通常是：

1. 主工作区保持在基线分支上。
2. 有新任务时，新建一个 worktree。
3. 该任务只在这个目录里开发。
4. 当基线分支前进后，把这个 worktree 同步到最新基线。
5. 任务完成后集成对应分支。
6. 集成完成后移除这个 worktree。

这是最推荐的默认方式，因为边界清晰，也最不容易误用。

### 基于槽位的工作流

在这种方式里，你会保留少量固定名字的 worktree 目录，反复复用。

典型结构：

```text
repo/
├── .worktrees/
│   ├── slot-1/
│   └── slot-2/
└── <主工作区>
```

适合这些情况：

- 编辑器、工具链或 agent 依赖稳定目录
- 一个任务做完后，常常会立刻在同一个位置开始下一个任务
- 你能接受在复用前先确认这个槽位已经清理干净

工作流通常是：

1. 主工作区保持在基线分支上。
2. 选择一个空闲槽位，比如 `slot-1`。
3. 当前任务在这个槽位里进行。
4. 当基线分支前进后，把这个槽位同步到最新基线。
5. 任务集成完成后，把槽位清理出来。
6. 下一个任务继续复用这个槽位。

这种方式更高效，但也更依赖纪律，因为目录名本身不再直接表达任务内容。

## 应该选哪一种

如果你更看重清晰度，选基于任务的工作流。

如果你更看重路径稳定，选基于槽位的工作流。

如果没有明确偏好，优先使用基于槽位的工作流。

## 具体操作步骤

以下是一个任务从开始到结束的完整流程。

### 1. 在主工作区开始

确认你当前在主工作区目录中，且处于基线分支（`main`、`develop` 等）并已拉取到最新。

### 2. 让 agent 创建 worktree

用自然语言告诉 agent 你要做的任务：

- `帮我为 login-page 任务新建一个 worktree。`
- 槽位式：`把 slot-1 分配给一个叫 login-page 的新任务。`

agent 会创建 `.worktrees/login-page/`（或 `.worktrees/slot-1/`），并在里面建好对应的任务分支。

### 3. 打开 worktree 目录

用 IDE 或 AI 工具打开新建的 worktree 目录。每个 worktree 都是一个完全独立的工作目录——当作打开了项目的另一份副本来用就行。

- VS Code: `code .worktrees/login-page/`
- Cursor 或其他 AI IDE: 打开 `.worktrees/login-page/` 文件夹
- CLI agent: `cd .worktrees/login-page/` 并在这里启动新的 agent 会话

### 4. 在 worktree 中开发

所有开发工作都在这个目录里进行，按正常节奏提交代码。主工作区完全不受影响。

如果开发期间基线分支有更新，同步一下：

- `把 login-page 的 worktree 同步到 develop。`

### 5. 合并并清理

任务完成后，将任务分支合并回基线分支（通过 PR、merge commit 或团队约定的方式），然后清理：

- `login-page 已经合并了，把它的 worktree 清理掉。`

agent 会移除 `.worktrees/login-page/` 并删除已合并的任务分支。如果是通过远程 PR 合并的，记得在主工作区拉取一下基线分支再开始下一个任务。

### 槽位式的差异

如果你选择了基于槽位的工作流，整体还是同样的六步，但其中几步的行为不同：

| 步骤 | 任务式 | 槽位式 |
|------|-------|-------|
| **2. 创建** | 每个任务新建一个目录：`.worktrees/login-page/` | 复用固定目录：`.worktrees/slot-1/` |
| **3. 打开** | 打开以任务命名的目录 | 每次打开同一个槽位目录——IDE 书签和工具配置始终有效 |
| **5. 清理** | 整个 worktree 目录一起移除 | 保留目录，但将槽位重置为干净状态，准备给下一个任务使用 |

槽位式的对话示例：

```text
你：   "把 slot-1 分配给一个叫 login-page 的新任务。"
Agent: → 复用 .worktrees/slot-1/，创建分支 login-page

       （在 slot-1 中正常开发）

你：   "login-page 已经合并了，把 slot-1 清理出来准备复用。"
Agent: → 将 slot-1 从已合并的分支上脱离，槽位恢复空闲
```

## 多 worktree 并行开发

以上步骤描述的是单个任务。真正的威力在于同时进行多个任务。

创建多个 worktree，分别在不同的 IDE 窗口或 agent 会话中打开：

```text
repo/
├── .worktrees/
│   ├── login-page/    ← Agent A 在这里工作
│   └── fix-timeout/   ← Agent B 在这里工作
└── <主工作区>          ← 停留在基线分支，用于同步和集成
```

因为每个 worktree 拥有独立的目录和独立的分支，多个 agent 可以同时开发而互不干扰——不存在文件级冲突、锁竞争或意外覆盖。

每个 agent 各自独立地执行上面的第 3–6 步。任务合并后及时清理对应的 worktree 即可。

## 最佳实践

- 主工作区尽量保持干净，并贴近最新基线分支。
- 一个 worktree 只做一个任务——不要在同一个 worktree 里混合不相关的工作。
- 多任务并行时，优先用基于任务的方式。
- 依赖稳定路径时，优先用基于槽位的方式。
- 已完成的 worktree 及时清理，避免分支归属变混乱。
